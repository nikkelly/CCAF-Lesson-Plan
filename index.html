~8–10 hrs · Do not rush

One course, 84 lectures, 10 quizzes. This is your primary textbook — it covers Domains 3 (Prompt Engineering, 20%) and 5 (Context Management, 15%) almost entirely, and underpins everything else. The GitHub notebooks are your hands-on companion; run them while watching the lectures.

Building with the Claude API core
84 lectures across API auth, conversations, tool use, RAG, streaming, structured output, multimodal, and architectural patterns.
8+ hrs ~35% of exam

Must-master topics

  • PRECISE framework — Persona, Role, Explicit instructions, Context, Instructions, Steps, Examples. Know what breaks when you omit each component
  • Structured output / JSON schemas — enforcing schemas with tool_choice, validation retry loops, graceful degradation when parsing fails
  • Tool use mechanics — stop_reason: "tool_use" vs "end_turn", appending tool results to conversation history, multi-turn tool loops
  • System prompt architecture — ordering rules, persona before context, XML tags for structure
  • Few-shot prompting — example selection, ordering matters (recency bias), what makes a good negative example
  • Prompt caching — cache_control breakpoints, what's cacheable, cost impact, when not to cache
  • Conversation compaction — summarization patterns, context window budget management
  • RAG patterns — retrieval-augmented generation integration, when to use vs. long-context
  • Streaming — when to stream vs. single response; latency tradeoffs for agentic systems
  • Message Batches API — async processing of large request volumes at reduced cost; exam tests when it's appropriate (bulk independent tasks) vs. inappropriate (real-time, dependent tasks)
  • "Lost in the middle" effect — models reliably process content at start and end of long inputs; findings in the middle get missed. Mitigation: key summaries at top, explicit section headers, trim verbose tool outputs before accumulation

Exam anti-patterns (Domain 3)

  • No Examples section in PRECISE → tone/length variance in production
  • Using natural language output where JSON schema would ensure reliability
  • Retry loops without exponential backoff or max-attempt caps
  • Caching content that changes frequently (user-specific context)
  • Prompt injection vulnerabilities in unsanitized user inputs passed to system prompt

Exam anti-patterns (Domain 5)

  • Not setting cache_control breakpoints on long, stable system prompts
  • Letting context windows fill without summarization or compaction
  • Passing entire conversation history to every subagent (context isolation failure)
⚑ Exam signal: "Responses vary in tone and length" → missing Examples in PRECISE. "Output unreliable across sessions" → missing structured output constraints.

Phase 03 MCP — Tool Design & Integration
~3–4 hrs · Domain 4 (18%)

Two courses back to back. You've wired MCP servers in Claude Code already — use this phase to formalize the primitives and learn production anti-patterns. The exam presents broken or suboptimal tool schemas and asks you to identify the flaw. Building a minimal MCP server during this phase is more valuable than extra video time.

Introduction to Model Context Protocol exam: Domain 4
MCP primitives (tools, resources, prompts), building servers and clients in Python from scratch.
~1.5 hrs exam weight

Topics to focus on

  • Three primitives: tools, resources, prompts — know when each is appropriate; exam tests tool vs. resource boundary (read-only data = resource, not tool)
  • JSON Schema for tool inputs — required fields, types, descriptions; spot malformed schemas
  • Tool description quality — descriptions Claude uses to decide whether to call the tool; vague descriptions = misrouting
  • stdio vs. SSE transport — stdio for local subprocess, SSE for remote/cloud servers streaming over HTTP
  • MCP server vs. client roles — who exposes capabilities, who consumes them
  • Scoping: project-level .mcp.json vs. user-level ~/.claude.json and when to use each

Exam anti-patterns

  • Exposing content catalogs as tools instead of resources — wastes tokens on exploration
  • Vague tool descriptions — Claude will prefer built-in tools (Grep, Read) over MCP tools with weak descriptions
  • Using stdio transport for a cloud/remote MCP server
  • One giant "do everything" tool instead of scoped, single-purpose tools
⚑ Exam signal: Questions often show a tool schema and ask "what's wrong?" — malformed JSON Schema required fields and tool descriptions that don't explain output are the most common flaws.

MCP: Advanced Topics exam: Domain 4
Sampling, notifications, file system access, transport mechanisms, and production security patterns.
~1.5 hrs exam weight

Topics to focus on

  • Authentication in production MCP — environment variable expansion for tokens, not hardcoded credentials

Phase 04 Agentic Systems — Highest Exam Weight
~2 hrs · Domain 1 (27%)

The most exam-critical phase. 27% of questions come from Domain 1 — agentic architecture and orchestration. Two Academy courses cover it, but the real exam prep here is sketching architectures for the 6 exam scenarios and drilling practice questions. This is where judgment matters most — two answer choices often both "work" technically, but one is architecturally superior.

Introduction to Subagents core · 27%
Multi-agent orchestration, context delegation, hub-and-spoke patterns, task decomposition, and lifecycle management.
~1 hr critical

Topics to focus on

  • Hub-and-spoke (coordinator → subagent) pattern — when to use vs. flat pipeline; coordinator handles decomposition and routing
  • Parallel vs. sequential subagent execution — parallel for independent tasks; sequential when output of one feeds next
  • Subagents do NOT inherit context — must pass context explicitly; this is one of the biggest exam traps
  • Minimal footprint design — each agent gets only the tools it needs for its role; no over-provisioning
  • Agentic loop lifecycle — observe → think → act → respond cycle; stop_reason patterns
  • Clarifying ambiguity before long autonomous runs — front-load clarification questions, don't start a 20-step task on an ambiguous goal
  • Recovery strategies — what to do when a subagent fails mid-task (escalate, retry with context, surface partial results)
  • Human-in-the-loop interrupt design — when to pause for approval vs. proceed autonomously

Exam anti-patterns

  • "Super agent" pattern — one agent trying to do everything instead of hub-and-spoke
  • Subagent assumes it has parent context — must be explicitly passed
  • Automatically retrying with elevated permissions after a permission error — violates minimal footprint
  • No escalation path — agents that never ask for human input in production
  • Silently returning incomplete results after subagent failure
⚑ Exam signal: "Agent X is doing Y in production and misbehaving — what is the most likely architectural cause?" → Check context isolation first, then tool scoping, then missing escalation paths.

Agent SDK & Hooks — Self-Study core · Domain 1
No Academy course covers this directly — but stop_reason logic, the Task tool, lifecycle hooks, forkSession, and programmatic enforcement are heavily tested. This is docs + hands-on work.
~2 hrs critical gap

Agent SDK — must-know topics

  • stop_reason: "tool_use" vs "end_turn" — the agentic loop pivot; exam tests correct loop control flow. Never parse natural language to determine termination — always check stop_reason
  • Task tool — how the coordinator spawns subagents; must be in allowedTools; subagents cannot include Task in their own tools (no recursive spawning)
  • forkSession / --fork-session — copies conversation history into a new session ID; the original is unchanged; used for parallel exploration from the same checkpoint
  • session resumption — --continue or --resume appends to same session; requires capturing session_id and passing it explicitly
  • max_budget_usd — cost guard on query(); set from the start in automated pipelines, not after the bill arrives
  • Each subagent gets its own context window — context isolation is not automatic, it must be designed

Hooks — must-know topics

  • PreToolUse hook — intercepts before a tool is called; use for compliance enforcement, permission checks, financial guardrails. This is "programmatic enforcement" — deterministic, not probabilistic
  • PostToolUse hook — runs after tool result returns; use for data normalisation, audit logging, compliance checks on output
  • SessionStart hook — fires on startup/resume/clear/compact; use for context injection and setup
  • Stop hook — fires before Claude ends a session; use to verify task completion before closing
  • Programmatic vs. prompt-based enforcement — the single most important mental model on the exam: hooks/gates are deterministic; prompt instructions are probabilistic. For business-critical rules (refund limits, security checks), always use hooks, never rely on prompt instructions alone

Exam anti-patterns

  • Parsing Claude's natural language response to detect loop termination — check stop_reason instead
  • Using prompt instructions to enforce a $500 refund limit — use PreToolUse hook (deterministic)
  • Giving an agent 18 tools — 4–5 max for reliable tool selection
  • Sentiment-based escalation logic — sentiment ≠ complexity; use explicit criteria
  • Including Task in a subagent's allowedTools — subagents cannot spawn further subagents

The key mental model

⚑ Programmatic over prompts: Any time an exam answer offers "add prompt instructions" vs. "use a PreToolUse hook" for a business-critical rule — the hook is always correct. Prompts are probabilistic; hooks are guaranteed. This is the most common smart distractor on Domain 1.

Actions — docs-first, then build

Hooks docs (all hook types) Agent SDK docs How Claude Code works (stop_reason) Session management (fork/resume)
Hands-on task: Add a PreToolUse hook to any project that logs every Bash call before execution. Then make it block a specific command pattern. You'll understand programmatic enforcement in 20 minutes of doing this vs. an hour of reading about it.

Claude Code in Action core · Domain 2
CLAUDE.md hierarchy, plan mode, slash commands, Explore-Plan-Code-Commit workflow, Git/CI integration, tool permission allowlists.
~1 hr core

Configuration topics to nail

  • Settings file hierarchy — global (~/.claude/settings.json) vs. project (.claude/settings.json) vs. local (.claude/settings.local.json); local is never committed
  • allowedTools in settings.json (committed) — the right place for team-wide Bash command allowlists; NOT settings.local.json, NOT CLAUDE.md
  • CLAUDE.md inheritance — project root → subdirectory; more specific files override less specific; understanding the composition model
  • .claude/rules/ directory — the correct way to split a monolithic CLAUDE.md into topic-specific files (testing.md, api-conventions.md, deployment.md) with YAML frontmatter for path-scoping; a direct sample question topic
  • Session commands — /compact (manual context compaction), /context (show token usage), /clear (fresh context), /resume (reopen session), /memory (view auto-memory). Know when to use each; exam tests when /compact is right vs. /clear
  • -p flag + --output-format json — how to run Claude Code non-interactively in CI/CD pipelines; the correct pattern for CI integration
  • CI/CD non-interactive flags — running Claude Code in pipelines requires --no-confirm or equivalent; interactive mode in CI is a bug, not a feature
  • Plan mode — when to use (ambiguous specs, large refactors) vs. when it's overhead (clear, scoped tasks)
  • Custom slash commands — structure, scoping, when to define project-level vs. global

Exam anti-patterns

  • Tool permission allowlists in settings.local.json — won't apply to other developers
  • Team-wide permissions in global ~/.claude/settings.json — won't persist across machines
  • Using CLAUDE.md for tool permission enforcement — it's behavioral guidance, not enforced policy
  • Running Claude Code interactively in CI/CD — blocks pipelines, incorrect pattern
  • "Always use plan mode" — exam expects judgment, not blanket rules
  • Monolithic 500-line CLAUDE.md — correct answer is .claude/rules/ with path-scoped topic files
  • Using /compact when a clean start is needed — /clear is correct; /compact keeps (and can lose) old context
⚑ Exam signal: "CLAUDE.md grown to 500 lines, Claude misses sections" → split into .claude/rules/ files with path-scoped frontmatter. This is a published sample question.

Reference Mental Models Cheatsheet
Memorize these

The exam repeatedly tests a small set of architectural judgment calls. Internalize these and wrong answers become obviously wrong.

🔑The 7 Core Mental Models
  • Programmatic > prompts for critical rules. Hooks are deterministic; prompts are probabilistic. PreToolUse for guardrails, not system prompt text.
  • No implicit context. Subagents never inherit coordinator history. Pass everything explicitly.
  • Minimal footprint. Each agent/subagent gets only the tools it needs for its role. Not "all tools just in case."
  • stop_reason drives loops. Check stop_reason === "tool_use", never parse response text to detect continuation.
  • Resources ≠ tools. Read-only catalog data = MCP resource. Actions and writes = MCP tool.
  • Proportional solutions. If fixing tool descriptions solves it, don't build a routing classifier. The exam penalizes over-engineering.
  • Structured output ≠ semantic correctness. JSON schema prevents invalid JSON. It does NOT prevent hallucinated values. Know where each helps.
⚙️Config Location Quick Reference
  • Team-wide tool allowlists → .claude/settings.json (committed)
  • Personal/machine overrides → .claude/settings.local.json (never committed)
  • Global all-projects defaults → ~/.claude/settings.json
  • Project behavior guidance → CLAUDE.md (root or subdirectory)
  • Topic-scoped rules (large projects) → .claude/rules/topic.md with YAML frontmatter globs
  • CI/CD non-interactive → claude -p "task" --output-format json
  • MCP project-scope → .mcp.json (committed)
  • MCP user-scope → ~/.claude.json
🪝Hook Decision Tree
  • Before tool call fires → PreToolUse (block, log, transform input)
  • After tool call returns → PostToolUse (normalise, audit, check output)
  • Session starts/resumes → SessionStart (inject context, run setup)
  • Before session ends → Stop (verify task complete before closing)
  • Context approaching limit → PreCompact (back up state before compaction)
🔄Session Command Quick Reference
  • /compact — summarise conversation, reduce tokens, keep working in same session
  • /clear — fresh context window, same session ID, lose conversation history
  • /resume — reopen a previous session by ID from the picker
  • /context — show what's consuming token budget (diagnose context bloat)
  • /memory — view and edit auto-memory that persists across sessions
  • --fork-session — copy history into new session ID for parallel exploration

🧠The 2-Minute-Per-Question Rule
  • Most questions have one obviously wrong answer and one "almost right" distractor. Eliminating the two clearly wrong answers first gets you to a 50/50 much faster.
  • When two answers both "work technically" — the more conservative, minimal-footprint, or programmatic option is almost always correct.
  • If you're uncertain, flag and return. Don't spend 5 minutes on one question.
  • Domain 1 questions (27%) are the most judgment-intensive — give them slightly more time.
  • Domain 5 questions (15%) are often the fastest — once you know CALM and "lost in the middle," they're straightforward.
